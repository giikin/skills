# 环境变量配置规范

本文档定义了项目的环境变量配置规范，包括文件结构、配置原则和多环境管理策略。

## 配置文件结构

项目采用分层环境配置，通过 `.env` 文件和环境特定文件实现配置管理。

```
.env                        # 通用默认配置（所有环境共享）
.env.development            # 开发环境配置（本地开发）
.env.production             # 生产环境配置（正式环境）
.env.test                   # 测试环境配置（测试服务器）
.env.giiktok.production     # giiktok 生产环境配置（租户平台）
```

## 配置分层原则

### DRY 原则（Don't Repeat Yourself）

**核心思想**：通用配置只在 `.env` 中定义一次，各环境文件只覆盖不同的值。

**配置优先级**：环境特定文件 > `.env` 通用文件

### 配置分类

#### 放在 `.env` 的配置

1. **所有环境都相同的配置**
   - 独立第三方服务的完整域名
   - 默认的平台配置
   - 通用的前端页面路径

2. **大多数环境相同的配置**
   - 只有 1-2 个环境不同的配置项
   - 作为默认值，在特殊环境中覆盖

#### 放在环境特定文件的配置

1. **环境独有的配置**
   - 只在某个环境使用的配置项
   - 如：CDN 地址、特殊的构建配置

2. **覆盖默认值的配置**
   - 与 `.env` 中默认值不同的配置项

## 域名规则

项目支持多个部署域名，不同环境对应不同域名：

| 环境               | 域名                            | 说明                         |
| ------------------ | ------------------------------- | ---------------------------- |
| development        | 本地代理                        | 开发环境使用代理或测试服务器 |
| production         | https://manage.giikin.com       | 吉客印生产环境               |
| test               | https://manage-test.giikin.cn   | 测试环境                     |
| giiktok.production | https://manage.kingkungmeta.com | 吉客拓客生产环境             |

**开发环境特别说明**：

在开发环境（`npm run dev`）中，所有以斜杠开头的相对路径（如 `/third/gstore/`、`/fe/gstore/`）会自动通过 Vite 的代理配置转发到测试服务器，无需手动配置完整的测试服务器地址。这样可以：

- 避免跨域问题
- 统一管理后端服务地址（继承 .env 的默认配置）
- 方便切换不同的测试环境

### 相对路径 vs 完整 URL

**相对路径**（如 `/third/gstore/`）：

- 会自动使用当前部署域名
- 适用于部署在同一域名下的服务
- **开发环境**：斜杠开头的相对路径会自动配置代理到测试服务器
- 示例：`/third/gstore/` 在 giiktok 环境会访问 `https://manage.kingkungmeta.com/third/gstore/`

**完整 URL**（如 `https://gassapi.giikin.cn`）：

- 直接访问指定的独立服务
- 适用于独立部署的第三方服务
- 示例：`VITE_OLD_GASS_API_URL=https://gassapi.giikin.cn`

## 配置管理规范

### 添加新配置

1. **判断配置类型**
   - 所有环境都相同？→ 添加到 `.env`
   - 大多数环境相同？→ 添加到 `.env`，在特殊环境覆盖
   - 环境特有？→ 只添加到对应环境文件

2. **命名规范**
   - 使用 `VITE_` 前缀（Vite 项目要求）
   - 使用全大写 + 下划线分隔（`VITE_BASE_API_URL`）
   - 命名要清晰描述用途

3. **注释说明**
   - 在 `.env` 中添加清晰的分组注释
   - 独立服务要标注说明

### 修改现有配置

**修改前必须检查**：

1. **确认配置用途**：是否为独立服务？影响哪些环境？
2. **检查所有环境文件**：确保修改后各环境配置正确
3. **验证相对路径**：如果从完整 URL 改为相对路径，确认目标域名是否正确

**常见错误示例**：

```bash
# 错误：将独立服务改为相对路径
# 原配置：VITE_BASE_COUPANG_SELECTION_API=https://gassapi.giikin.cn/coupang
# 错误改为：VITE_BASE_COUPANG_SELECTION_API=/coupang
# 问题：域名从 gassapi.giikin.cn 变成了当前部署域名
```

## 多环境构建

### 构建命令

```bash
# 开发环境
npm run dev

# 生产环境
npm run build           # 使用 .env.production

# 测试环境
npm run build:test      # 使用 .env.test

# giiktok 生产环境
npm run build:giiktok   # 使用 .env.giiktok.production
```

### 环境变量加载顺序

Vite 加载环境变量的优先级：

```
.env.{mode}.local  >  .env.{mode}  >  .env.local  >  .env
```

实际项目中的加载：

1. 所有模式都加载 `.env`（通用配置）
2. 特定模式加载对应的 `.env.{mode}`（覆盖通用配置）

## 最佳实践

### 推荐做法

1. **配置精简化**：只在环境文件中写必要的覆盖配置
2. **注释清晰**：说明配置用途，特别是独立服务
3. **统一管理**：相同的配置只在 `.env` 中维护一次
4. **定期审查**：及时清理不再使用的配置项

### 避免做法

1. **重复配置**：在多个环境文件中重复相同的配置
2. **硬编码**：在代码中硬编码环境相关的 URL 或配置
3. **随意修改**：不理解配置用途就修改，特别是完整 URL
4. **混乱命名**：使用不规范的变量名
